## 验收场景

### RD-002: wait_for_result — XML `<task_result>` 解析 PASS

| 项目 | 详情 |
|------|------|
| **操作** | 在终端中执行 node 命令输出 `<task_result>` XML |
| **输入** | `node -e "process.stdout.write('<task_result><status>PASS</status><summary>All checks completed successfully</summary><details><test_count>42</test_count><passed>42</passed></details></task_result>')"` |
| **调用** | `wait_for_result` (timeoutMs=10000) |
| **结果** | `{"status":"PASS","summary":"All checks completed successfully"}` |
| **验证点** | XML 正确解析，status=PASS，summary 提取成功 |

### RD-004: wait_for_pattern 超时 + 快照 PASS

| 项目 | 详情 |
|------|------|
| **操作** | 发送 `wait_for_pattern` 查询一个永远不会匹配的模式 |
| **输入** | pattern=`NEVER_APPEAR_PATTERN_XYZ`, timeoutMs=5000 |

| **验证点** | 正确返回 timedOut=true，elapsedMs 合理（5232ms ≈ 5000ms 超时 + 轮询开销），snapshot 包含终端当前内容 |

### RD-006: content_only 过滤 + filter metadata PASS

| 项目 | 详情 |
|------|------|
| **操作** | 对比 raw+cleanAnsi tail 读取与 content_only 模式 |
| **调用 A** | `read_terminal` mode=tail, raw=true, cleanAnsi=true, tailLines=20 |
| **结果 A** | Returned Chars: 867, Original Chars: 1077, Raw Mode Filter: tail |
| **调用 B** | `read_terminal` mode=content_only, adapter=generic |
| **结果 B** | Returned Chars: 868, Original Chars: 868, Filter: Mode=content_only, Adapter=generic, Confidence=high, Removed=0, Critical Lines Preserved=4 |
| **验证点** | content_only 返回 filter 元数据（Mode, Adapter, Confidence, Removed, Critical Lines Preserved），关键字行保留 |

## 脱敏说明

- 本报告中无真实 session ID、token 或 API key
- 所有文件路径使用项目相对路径或通用描述
- terminalId 使用 MCP 工具返回的临时值，已随终端销毁失效
- REST API 端口 3098 为临时测试端口

## 限制

1. **Windows ConPTY**: 部分测试场景（如 REST API 创建终端后的 wait-result）在 Windows standalone 模式下受 ConPTY 限制，这些场景由 Jest 集成测试 (`rest-api.test.ts`) 补充覆盖
2. **Web UI 深度交互**: 完整的浏览器自动化测试需要 Playwright 环境，本次仅验证了页面可访问性和基本 UI 元素
3. **MCP 工具测试**: 所有 MCP 工具调用通过 persistent-terminal-mcp 自身的 MCP 服务器完成，形成了自举验证

## 结论

5/5 场景通过。v1.2.2 (构建版本 1.2.3) 的核心功能（wait_for_result 解析、wait_for_pattern 超时、content_only 过滤元数据、REST API 端点、Web UI 基本功能）均已通过真实 CLI 驱动验证。
