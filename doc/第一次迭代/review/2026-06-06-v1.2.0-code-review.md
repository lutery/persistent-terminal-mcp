## Findings

### [P0] `statusFile` 协作状态源没有真正接入 `get_terminal_status`

需求和设计已确认“`statusFile(JSON)` 优先，其次 XML `<task_result>`，最后才是 PTY 启发式”，并要求协作状态源命中时把 `semanticStatusConfidence` 标记为 `cooperative`。当前实现只是在创建阶段保存了 `session.statusFile`，后续状态查询完全没有读取该文件。

- 设计要求: [2026-06-06-system-design.md](../doc/2026-06-06-system-design.md) 第 260-278 行、第 436-437 行
- 实现仅保存路径: [src/terminal-manager.ts](/F:/Projects/OpenSrouce/WebApp/persistent-terminal-mcp/src/terminal-manager.ts:953) 第 963-967 行
- 状态查询仍然全部走启发式: [src/terminal-manager.ts](/F:/Projects/OpenSrouce/WebApp/persistent-terminal-mcp/src/terminal-manager.ts:568) 第 589-653 行
- `StatusProvider` 虽然存在，但 `TerminalManager`/MCP/REST/Web UI 没有调用链接入

影响：

- `get_terminal_status` 无法兑现本轮最关键的精确状态来源设计。
- `statusFile` 相关负向场景在验收计划中被要求覆盖，但当前实现实际上没有该功能路径。

### [P0] `ResultParser` 依赖 `fast-xml-parser`，但发布清单没有声明这个依赖

`src/result-parser.ts` 直接从 `fast-xml-parser` 导入 `XMLParser`，但 `package.json` 的 `dependencies` 中没有该包。当前工作区可能因为历史安装残留而掩盖了问题，但在干净环境执行 `npm install` 或作为库被消费时，`wait_for_result` / `ResultParser` 会直接失效。

- 代码导入: [src/result-parser.ts](/F:/Projects/OpenSrouce/WebApp/persistent-terminal-mcp/src/result-parser.ts:1)
- 依赖缺失: [package.json](/F:/Projects/OpenSrouce/WebApp/persistent-terminal-mcp/package.json:102) 第 102-110 行
- 开发报告却声称“新增依赖 fast-xml-parser”: [2026-06-06-v1.2.0-dev-report.md](../dev/2026-06-06-v1.2.0-dev-report.md) 第 156 行附近

影响：

- 打包结果不可复现。
- 真实环境一旦不是沿用当前 `node_modules`，该功能会在运行时崩溃。

### [P1] `read_terminal` 的过滤元数据在实现里被丢弃，偏离已确认的返回契约

系统设计明确要求 `content_only` / `last_response` 响应中包含 `filter` 元数据，但 `TerminalManager.readFromTerminal()` 虽然计算了 `metadata`，最终没有放进返回对象；MCP 工具也只返回纯文本，没有 `structuredContent`。

- 设计要求 filter metadata: [2026-06-06-system-design.md](../doc/2026-06-06-system-design.md) 第 260-278 行、452 行
- `content_only` 计算后丢弃 `metadata`: [src/terminal-manager.ts](/F:/Projects/OpenSrouce/WebApp/persistent-terminal-mcp/src/terminal-manager.ts:352) 第 369-377 行
- `last_response` 计算后丢弃 `metadata`: [src/terminal-manager.ts](/F:/Projects/OpenSrouce/WebApp/persistent-terminal-mcp/src/terminal-manager.ts:381) 第 395-403 行
- MCP `read_terminal` 仅回文本: [src/mcp-server.ts](/F:/Projects/OpenSrouce/WebApp/persistent-terminal-mcp/src/mcp-server.ts:540) 第 544 行、666-672 行

影响：

- 主代理无法获知过滤置信度、移除行数、关键行保留统计。
- REST/Web UI 也拿不到这部分结构化信息，无法兑现同一套设计契约。

### [P1] 真实驱动报告把可恢复会话标识和本机路径写进了版本库

需求和验收文档要求不要把敏感信息提交到仓库。当前真实驱动报告虽然没有直接暴露 API key，但提交了可恢复的 Claude 会话标识，以及本机代理脚本绝对路径。这类运行态标识至少属于敏感操作信息，不应进入 git 跟踪文档。

- 本机路径和启动命令: [2026-06-06-v1.2.0-real-cli-driver-report.md](../test/2026-06-06-v1.2.0-real-cli-driver-report.md) 第 10-16 行
- 可恢复 session id: 同文件第 79-80 行
- 验收计划要求文档与测试夹具使用假值或脱敏值: [2026-06-06-test-acceptance-plan.md](../test/2026-06-06-test-acceptance-plan.md) 第 142-145 行

影响：

- 文档副本传播后会暴露本地环境结构。
- 如果上游 CLI 对 session id 具备恢复能力，这相当于把会话恢复凭据放进仓库。

### [P2] 对外版本和类型元数据存在漂移

这不是最核心的行为缺陷，但会误导调用方。

- MCP server 自报版本仍是 `1.0.0`: [src/mcp-server.ts](/F:/Projects/OpenSrouce/WebApp/persistent-terminal-mcp/src/mcp-server.ts:41) 第 43-45 行

影响：

- 使用导出类型的上层调用方会得到错误的编译提示。
- 服务元数据与实际版本不一致，不利于客户端诊断和缓存。

## 审查结论

本轮不建议放行。优先级顺序应为：

1. 补齐 `statusFile -> get_terminal_status` 的协作状态接入
2. 修正 `fast-xml-parser` 依赖声明并在干净环境复测
3. 让官方测试入口覆盖新增场景，并把 Windows 目标环境跑到稳定可复核
4. 修正 Web UI 假覆盖与 `read_terminal` 过滤元数据缺失
5. 清理真实驱动报告中的敏感运行态信息并重做脱敏
