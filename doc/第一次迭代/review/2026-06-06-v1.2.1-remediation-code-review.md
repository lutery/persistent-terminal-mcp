## Findings

### [P0] `resume_terminal` 存在命令注入风险

**证据**:

- `src/terminal-manager.ts:1125-1128`
- `src/mcp-server.ts:1242-1258`
- `src/rest-api.ts:474-491`
- `src/web-ui-server.ts:384-396`

`resumeTerminal()` 直接把外部传入的 `sessionId` 拼接成 shell 命令：

```ts
const resumeCommand = `claude --resume ${options.sessionId}`;
```

`sessionId` 来自 MCP tool、REST API、Web UI 请求体，目前只校验了字符串存在，没有格式限制、shell 转义或参数化执行。由于该命令随后会作为初始化命令写入 PTY，在 Windows PowerShell/cmd 或类 Unix shell 中，带有 `&`、`;`、换行等控制字符的输入可以改变实际执行命令。

**影响**:

- 这是新增的远程输入到 shell 命令执行路径。
- REST/Web UI 场景下风险更高，因为 HTTP 请求体可直接到达该路径。
- 该问题不属于实现难度问题，而是可用性和安全边界问题，必须修复后才能通过审查。

**整改建议**:

1. 对 Claude session id 定义严格 allowlist，例如只允许 `[A-Za-z0-9._:-]` 且限制长度。
2. 拒绝包含空白、引号、重定向、管道、命令连接符、换行的 `sessionId`。
3. 在 MCP、REST、Web UI 和 `TerminalManager.resumeTerminal()` 四层至少保证核心层有同一套校验。
4. 增加单元测试和 HTTP 测试，覆盖 `abc&echo hacked`、`abc;echo hacked`、带换行、带引号等输入，预期返回 400 或抛出 validation error，且不会创建终端。

### [P1] 官方集成测试入口仍然失败，不能作为验收通过依据

**证据**:

- `package.json:63-73`
- 本轮执行 `npm run test:integration`，退出码 `1`
- 失败脚本: `test:integration:terminal`
- 输出摘要: `通过: 0`，`失败: 6`，多次 `Failed to create terminal: Error: File not found:`

整改已经把新增脚本接入 `test:integration`，但官方入口现在仍然红。开发报告中把 Windows PTY 限制作为解释，但当前 `package.json` 暴露的验收命令没有平台判断、跳过策略或替代命令说明，用户或 CI 执行 `npm run test:integration` 会得到失败。

**影响**:

- “新增集成脚本已接入”成立。
- “集成验收通过”不成立。
- 这会继续造成测试报告和实际命令结果不一致。

**整改建议**:

1. 明确区分跨平台集成测试和 PTY 平台相关测试。
2. 让 `npm run test:integration` 在 Windows 上不执行已知不可用脚本，或让脚本自身以可解释的 skip 状态退出 0。
3. 保留一个严格命令，例如 `npm run test:integration:pty`，用于支持真实 PTY 的环境。
4. 更新开发报告，不能再把失败的官方入口描述为已完成验收。

### [P1] 测试存在假绿风险：Windows 下直接 `return` 被 Jest 计为通过

**证据**:

- `src/__tests__/web-ui-server.test.ts:80-84`
- `src/__tests__/terminal-manager.test.ts:9-13`
- `src/__tests__/terminal-manager.test.ts:545-749`

`ptyTest()` 在 Windows 或 PTY 不可用时直接 `return`：

```ts
test(name, async () => {
  if (!ptyAvailable) return;
  await fn();
});
```

这种写法会让 Jest 把测试计为 passed，而不是 skipped。开发报告中写到 `154/154` 通过，但其中一部分 PTY 行为没有执行。这不是“跳过”，而是“未执行但计为通过”。

**影响**:

- 单测绿灯不能证明 PTY 相关能力在 Windows 下通过。
- `statusFile` 的 cooperative 路径虽然不需要真实 PTY，但当前新增测试也放了 `if (IS_WINDOWS) return`，导致 Windows 下连纯内存 fake session 的 cooperative 路径也没有执行。

**整改建议**:

1. 使用 `describe.skip` / `test.skip` / 条件选择 `const maybeTest = condition ? test : test.skip`，让报告真实显示 skipped。
2. 对不依赖 PTY 的测试移除 Windows return，例如 `statusFile` fake session 测试。
3. 报告中区分 passed、failed、skipped，不再写总数全 passed 来替代真实覆盖。
