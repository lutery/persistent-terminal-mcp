# v1.2.0 真实驱动验收缺口分析

- 日期: 2026-06-06
- 输入材料: [2026-06-06-v1.2.0-real-cli-driver-report.md](../test/2026-06-06-v1.2.0-real-cli-driver-report.md)
- 结论: `验收证据不足，不能作为通过依据`

## 观察到的缺口

1. 新增 MCP 能力没有在真实驱动里完成闭环验证。
   - `wait_for_result` 未测
   - `wait_for_pattern` 超时快照未测
   - `content_only` 未测

2. REST API 和 Web UI 只做了“基础功能正常”的口头结论，没有跑新增 surface 的完整操作链。

3. 报告中的关键结论依赖“需要重启 MCP 客户端后再测”，说明验收环境与被测构建并未真正同步。

4. 报告写入了真实 session id 与本地代理脚本路径，不符合脱敏要求。

## 根因分析

### 根因 1: 验收前没有强制校验运行中的 MCP 构建版本

报告第 35-37 行已经说明运行中是旧构建，因此后续关于 `wait_for_result`、`content_only`、`wait_for_pattern` 的“未测试”并不是功能本身的通过或失败，而是测试前置条件没有满足。

这属于验收流程问题，不是“补充说明”能掩盖的。

### 根因 2: 官方测试入口没有收敛到本轮能力

`package.json` 的 `test:integration` 仍然串的是旧脚本，没有把迭代 1 的五个新脚本纳入执行路径。研发报告虽然列出了这些脚本，但仓库默认入口没有复用它们，导致“开发完成”和“可复核验收”脱节。

### 根因 3: Surface-specific 测试设计不足

Web UI 测试没有真正起服务、发请求、连 WebSocket。真实驱动报告也没有补足这一缺口，所以最终形成了“代码看起来有功能，但 surface 没有证据”的状态。

### 根因 4: 文档出厂前没有脱敏检查

真实驱动报告直接记录了会话恢复标识和本机绝对路径，说明缺少最基本的报告出厂检查步骤。

## 整改建议

### A. 先修实现偏差

1. 补齐 `statusFile` 协作状态读取链路
2. 修正 `fast-xml-parser` 依赖声明
3. 给 `read_terminal` 增加 filter metadata 返回

### B. 再修验收流程

1. 在真实驱动开始前，先执行一次“运行中 MCP 版本校验”
2. 明确规定：如果被测构建未加载，整轮真实驱动无效，不能记为“部分通过”
3. 把新增五个集成脚本接入 `test:integration`
4. 为 REST API / Web UI 增加真正的 surface 级自动化验证

### C. 最后重跑真实驱动

最小重跑范围：

1. `RD-002 wait_for_result`
2. `RD-004 wait_for_pattern` 超时快照
3. `RD-006 content_only`
4. `RD-008 REST API` 新端点
5. `RD-009 Web UI` 新交互

## 建议的复验顺序

1. `npm install`
2. `npm run build`
3. 重启 MCP 客户端，确认加载的是新构建
4. `npm test`
5. `npm run test:integration`
6. 重跑真实 Claude Code CLI 驱动
7. 输出脱敏后的新报告

## 复验通过标准

1. 新增 MCP 工具和模式全部能被真实客户端调用到
2. REST API 新端点完成正向和负向验证
3. Web UI 完成状态展示、等待操作、恢复操作验证
4. 报告中不出现真实 session id、token、本机敏感路径
5. 官方自动化入口可以复现通过结果
