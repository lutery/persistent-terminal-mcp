## Findings

### [P1] 全仓仍存在 Windows 直接 `return` 的 Jest 假绿测试

**证据**:

- `src/__tests__/mcp-server.test.ts:21`
- `src/__tests__/mcp-server.test.ts:42`
- `src/__tests__/rest-api.test.ts:70`

本轮提交已经修复了 `src/__tests__/terminal-manager.test.ts` 中上一轮指出的 16 处问题：PTY 依赖测试改为 `ptyTest()`，fake-session/error-path 测试移除了 Windows guard，这一部分通过复核。

但全仓测试扫描仍存在以下模式：

```ts
if (process.platform === 'win32') return; // node-pty conpty issues on Windows
```

这些测试在 Windows 上仍会被 Jest 计为 `passed`，而不是 `skipped`。当前 `npx jest --runInBand` 输出 `31 skipped, 137 passed, 168 total`，但其中仍有至少 3 个 test body 在 Windows 上没有执行核心断言。研发总结中的“已验证测试代码中没有剩余的 if (IS_WINDOWS) return”只覆盖了变量名形式，没有覆盖 `process.platform === 'win32'` 形式。

**影响**:

- 测试报告仍不能准确表达真实覆盖状态。
- `mcp-server.test.ts` 的 active/terminated terminal 状态测试和 `rest-api.test.ts` 的 backward-compatible terminal creation 测试在 Windows 上仍是假通过。
- 与用户要求“测试代码没有虚假测试假绿”不一致。

**整改建议**:

1. 提供统一的 `ptyTest()`/`skipOnWindowsPty()` helper，并在所有 PTY 依赖 Jest 测试中使用 `test.skip`。
2. 对不依赖真实 PTY 的 REST/MCP 错误路径测试，改用 fake session 或 fake manager，让 Windows 也真实执行。
3. 新增或保留一个扫描验收命令，覆盖 `if (IS_WINDOWS) return` 和 `if (process.platform === 'win32') return` 两种形式。

### [P2] `package-lock.json` 版本未同步到 1.2.4

**证据**:

- `package.json:3` 为 `1.2.4`
- `package-lock.json:3` 仍为 `1.2.3`
- `package-lock.json:9` 仍为 `1.2.3`
- `src/mcp-server.ts:44` 为 `1.2.4`
- `src/rest-api.ts:532` 为 `1.2.4`
- `dist/mcp-server.js:20` 为 `1.2.4`
- `dist/rest-api.js:490` 为 `1.2.4`

本轮提交说明写到“Version bumped to 1.2.4 across package.json, mcp-server.ts, rest-api.ts”，但 `package-lock.json` 根包版本没有同步。虽然这不影响 `npm run build`，但会造成发布元数据和锁文件漂移。

**整改建议**:

1. 使用 `npm install --package-lock-only` 或等价方式同步 lockfile 根版本。
2. 将版本一致性扫描纳入发布前检查，至少覆盖 `package.json`、`package-lock.json`、MCP server、REST API、dist 和版本断言测试。
